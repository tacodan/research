# Customer Schema v1

This page documents the historical `customers_v1` schema described in the raw note. It should not be treated as proof of the current production schema.

## Historical Role

- Physical index name: `customers_v1`
- Search alias created for application access: `customers-search`
- Source note position: initial customer index creation flow

## v1 Field Set

| Field | Type | Notes |
| --- | --- | --- |
| `id` | `keyword` | Customer identifier from the primary database. |
| `num_service_addresses` | `integer` | Count of active service addresses. |
| `addy_1` | `text` + `keyword` | Searchable address line with exact-match sub-field. |
| `city` | `text` + `keyword` | Searchable city with exact-match sub-field. |
| `state` | `keyword` | Two-letter state or province code. |
| `sales_rep_id` | `keyword` | Assigned sales representative identifier. |
| `company_name` | `text` + `keyword` | Searchable company name with exact-match sub-field. |
| `last_updated` | `date` | ISO 8601 timestamp expected before indexing. |
| `status_state` | `keyword` | Primary customer status. |
| `status_state_sub` | `keyword` | Secondary customer status. |

## Mapping Strategy

- `keyword` fields are used for identifiers, exact-match filtering, sorting, and aggregations.
- `text` plus `.keyword` multi-fields are used where both full-text search and exact sorting/filtering are needed.
- `integer` is used for address counts.
- `date` is used for `last_updated`, with the source note expecting ISO 8601 formatting before indexing.

## Historical Creation Flow

1. Create `customers_v1` with the mapped fields above.
2. Create the `customers-search` alias and point it at `customers_v1`.
3. Verify the index, mapping, and alias in Dev Tools.

## Verification Commands From The Note

```json
GET /_cat/indices?v
GET /customers_v1/_mapping
GET /_cat/aliases?v
```
